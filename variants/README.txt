Варианты сборок VGM-плеера для A/B тестирования OPL3-лага
=========================================================

Базовая частота ISR: 2734 Hz (3500000 / 1280)
Базовый бюджет:      16 команд на тик

Обозначения:
  b<N>   — command budget (макс. кол-во команд между принудит. паузами)
  hf     — half frequency  (ISR 1367 Hz, вдвое реже)
  df     — double frequency (ISR 5468 Hz, вдвое чаще)

Файл                        Бюджет  Частота ISR  Примечание
───────────────────────────  ──────  ───────────  ──────────────────────────────
tsconf_v1_b8.img               8  +   2734 Hz     Маленький бюджет — частые паузы
tsconf_v2_b32.img             32  -   2734 Hz     Большой бюджет — реже паузы
tsconf_v3_b32_hf.img          32  +  1367 Hz     Большой бюджет + ISR вдвое реже
tsconf_v4_b16_hf.img          16  +  1367 Hz     Базовый бюджет + ISR вдвое реже
tsconf_v5_b8_df.img            8  -  5468 Hz     Маленький бюджет + ISR вдвое чаще
tsconf_v6_b16_df.img          16  -  5468 Hz     Базовый бюджет + ISR вдвое чаще
tsconf_v7_b24.img             24  -  2734 Hz     Средний бюджет
tsconf_v8_b64.img             64  -  2734 Hz     Очень большой бюджет

v9_b32_df (budget=32, ISR 5468 Hz) — не собрался (0 bytes free, overflow).

Что тестировать
───────────────
1. Если лаг пропадает на v2/v7/v8 — дело в бюджете (мало команд между паузами).
   → Увеличить VGM_FILL_CMD_BUDGET в финальной сборке.

2. Если лаг пропадает на v3/v4 (hf) — ISR слишком частый, не успевает.
   → Уменьшить частоту ISR.

3. Если лаг пропадает на v5/v6 (df) — ISR слишком редкий, грубый тайминг.
   → Увеличить частоту ISR.

4. Если ничего не помогает — проблема не в бюджете/частоте,
   а в чём-то другом (OPL3 задержка записи, VGM парсинг и т.д.)
