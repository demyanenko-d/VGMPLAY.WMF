# Wild Commander — Обзор Системы Плагинов

> Версия документа: 2.0 — дополнена по результатам анализа исходников  
> Источники: `inc/wc_api.h`, `asm/wc_api.s`, `asm/crt0.s`, `DiskRef/Help.txt`, `DiskRef/WC_History.txt`

## Введение

Wild Commander (WC) — файловый менеджер с системой плагинов для платформы
**TS-Configuration** (ZX Spectrum Evo). Выполняет роль основной оболочки и
загрузчика для работы с FAT32-дисками, SD-картами и IDE.

WC запускается из BootROM / vDOS и предоставляет:
- Двухпанельный интерфейс (80 × 25/30/36 TXT-режим)
- FAT32 (длинные и короткие имена до 255 символов)
- Систему плагинов (.WMF) с горячей загрузкой/выгрузкой
- Поддержку устройств: SD (Z-Controller), IDE NEMO, IDE SMUC, NeoGS SD
- Встроенный просмотрщик (F3): TXT, HEX, TGA, SCR, Gigascreen, MGS — до 1 МБ
- Мультиязычность (ENG / RUS, переключение CTRL+SHIFT)

## Общая концепция

Плагины WC (.WMF файлы) — модульные расширения, выполняемые в адресном
пространстве Z80. Поддерживают до **64 страниц** (1 МБ) кода/данных.

Типичные области применения:
- Просмотр файлов различных форматов (BMP, GIF, ILBM, SCR)
- Воспроизведение музыки и видео (VGM, MOD, WAV, MP3/GS)
- Редактирование файлов (TXT Editor с подсветкой синтаксиса)
- Работа с ROM/Flash (ROM_PROG)
- Монтирование образов дисков (MOUNTER: TRD, TAP, FDI)
- Утилиты (часы, скринсейвер, компилятор, дамп памяти)

## Типы запуска плагинов

При запуске плагина, в регистр **A** передается тип вызова:

| Код | Константа (C) | Описание |
|-----|---------------|----------|
| #00 | `WC_PCOND_EXT_ONLY` | По расширению файла (ENTER / F3) |
| #01 | `WC_PCOND_ON_LOAD` | При первой загрузке плагина (ENTER) |
| #02 | `WC_PCOND_TIMER` | По таймеру скринсейвера + меню F10 |
| #03 | `WC_PCOND_F10` | Из меню запуска плагинов (F10) |
| #04 | `WC_PCOND_VIEWER` | Из меню вьювера (SHIFT+F3 / SHIFT+ENTER) |
| #05 | `WC_PCOND_F10_VIEW` | F10 + SHIFT+F3 |
| #11 | `WC_PCOND_F10_F2` | F10 + F2 |
| #12 | `WC_PCOND_F2` | По нажатию F2 |
| #13 | `WC_PCOND_F10_F4` | F10 + F4 |
| #14 | `WC_PCOND_F4` | По нажатию F4 |
| #15 | `WC_PCOND_F10_V_F4` | F10 + SHIFT+F3 + F4 |

> **Примечание:** Все плагины также вызываются по расширению, если расширения
> указаны в блоке расширений заголовка (offset +64). ENTER и F3 разделены
> начиная с WC v1.00 — флаг +63 определяет, какой из них запускает плагин.

## Входные параметры при запуске

WC делает `CALL #8000`. В `crt0.s` (SDCC) входные регистры сохраняются
в глобальные переменные автоматически.

```asm
; Входные параметры:
A       - тип вызова (см. таблицу выше)
[DE,HL] - длина файла в байтах (DE=hi16, HL=lo16)
BC      - адрес строки с именем файла (для типов #00, #04, #14)
          формат: name(1-255), 0   (длинное FAT имя)
IX      - адрес структуры активной панели

; Альтернативные регистры (только для #00,#04,#12,#14):
A'      - номер расширения (0-31) для типа #00
DE'     - количество файлов/каталогов в панели (включая "..")
HL'     - номер объекта из панели (от 1)
```

В SDCC-плагине (`crt0.s` автоматически):
```c
extern uint8_t   wc_file_ext;   // ← A' (номер расширения)
extern char *    wc_file_name;  // ← BC (указатель на имя)
extern uint32_t  wc_file_size;  // ← [DE,HL] (32-бит размер)
```

## Коды возврата из плагина

При выходе из плагина регистр **A** определяет действие WC:

| A | Константа (C) | Действие |
|---|---------------|----------|
| 0 | `WC_EXIT_ESC` | Просто выход (ESC) |
| 1 | `WC_EXIT_UNRECOGNIZED` | Файл не опознан → передать другому плагину (тип #00) |
| 2 | `WC_EXIT_NEXT` | Перейти на следующий файл (тип #00) |
| 3 | `WC_EXIT_RELOAD_DIR` | Перечитать каталог |
| 4 | `WC_EXIT_PREV` | Перейти на предыдущий файл (тип #00) |

В SDCC-плагине: `wc_exit_code = 0; return 0;` — `crt0.s` считывает
`wc_exit_code` в A перед `RET` обратно в WC.

> **Код 2/4 (NEXT/PREV):** WC вызывает тот же плагин повторно с другим
> файлом. `crt0.s` заново обнуляет BSS (`_DATA`) и копирует инициализаторы
> (`_INITIALIZER → _INITIALIZED`), поэтому все `static` переменные сбросятся.

## Вызов API функций

Все функции WC вызываются через точку доступа #6006:

```asm
WLD     EQU #6006

; Пример вызова функции PRWOW (вывод окна)
PRWOW   LD A,1
        JP WLD

; Пример вызова ESC (проверка нажатия ESC)
ESC     LD A,#17
        JP WLD
```

**Регистровые соглашения WC API:**
- Вход: **A** = номер функции, остальные параметры в регистрах (см. справочник)
- Некоторые функции используют **A'** (альтернативный A через `EX AF,AF'`)
- **Портит**: AF, BC, DE, HL, IX
- **Сохраняет**: IY, AF', BC', DE', HL', SP

> ⚠️ **IX сохраняется** диспетчером P8000, но **AF, BC, DE, HL портятся**
> при каждом вызове `CALL #6006`.

## Работа со страницами памяти

### Логическая нумерация

Плагин запрашивает N страниц в заголовке (offset +34). WC выделяет их
из пула `#61–#A0`. Плагин обращается по **логическому** номеру 0..N−1.

### Окна CPU и функции маппинга

| Окно | Адреса | Функция ASM | Функция C | Примечание |
|------|--------|-------------|-----------|------------|
| Win 0 | `#0000–#3FFF` | `MNG0_PL` (#4E) | `wc_mng0_pl(page)` | ⚠ Данные для WC API нельзя класть сюда |
| Win 2 | `#8000–#BFFF` | `MNG8_PL` (#4F) | `wc_mng8_pl(page)` | Код плагина — обычно стартовая страница |
| Win 3 | `#C000–#FFFF` | `MNGC_PL` (#00) | `wc_mngc_pl(page)` | ⚠ Сбрасывается после КАЖДОГО вызова WC API |

Специальные значения для `MNGC_PL`:
- `#FF` — шрифтовая страница (нельзя писать в `#E000–#FFFF`!)
- `#FE` — первый текстовый экран (с панелями WC)

### Резидент перехода (#6020)

Переключает страницу плагина и делает **безвозвратный** JP:

```c
// C: никогда не вернётся в вызывающий код
void wc_resident_jump(uint8_t page, uint16_t addr);
```

```asm
; ASM: A = страница (0..N-1), HL = адрес (#8000–#BFFF)
        LD A,1
        LD HL,#8000
        CALL #6020      ; → WC переключает Win2 на стр.1, JP (HL)
```

### Резидент вызова (#6028)

Вызывает код в другой странице плагина **с возвратом**:

```c
void wc_resident_call(uint8_t page, uint16_t addr);
```

```asm
; ASM: A = страница, HL = адрес. Сохраняет всё кроме A,HL.
        LD A,2
        LD HL,#A000
        CALL #6028      ; → WC переключает, CALL (HL), возвращает стр. назад
```

> Добавлено в WC v1.09 (31.07.2024). До этого многостраничные плагины
> требовали прямой OUT в порт Page2 (#12AF).

## Важные замечания

1. **При входе в плагин может быть любой экран!** Обязательно вызывайте
   `GEDPL` (#0F) / `wc_gedpl()` для переключения на текстовый режим.

2. **Вызывать GEDPL лучше после анализа файла**, когда убедились что файл
   подходит для плагина (и вернули `wc_exit_code = 1` если нет).

3. **Обязательно сохраняйте IX** при входе в плагин, если планируете
   использовать файловые функции.

4. **⚠ Win3 (#C000) сбрасывается после каждого вызова WC API!**  
   Внутри `CALL #6006` WC переключает окно `#C000` на текстовую страницу
   и **не восстанавливает** его. Если плагин работал через `mngcvpl()` /
   `mngc_pl()`, он **обязан** повторить маппинг после любого UI-вызова
   (`gedpl`, `prwow`, `txtpr`, `rresb`, `load512` и т.д.).

5. **GEDPL затирает #1000–#1447 в окне 0** — нельзя хранить данные в этой области.

6. **Буферы и строки для файловых функций** (`load512`, `fentry`, `findnext`,
   и т.д.) **обязаны** быть в `#8000–#FFFF`. WC ремапит `#0000–#7FFF`
   под FAT-движок внутри каждого файлового вызова.

7. **FindNext портит #0000–#1FFF** на странице драйвера — не хранить
   данные там при итерации по каталогу.

8. **WC API портит регистры:** AF, BC, DE, HL, IX. Сохраняет: IY, AF',
   BC', DE', HL', SP. Это верно для **всех** функций через `CALL #6006`.

9. **Стек не настраивается** — плагин использует стек WC (в Win 1, `#4000–#7FFF`).
   Стек WC достаточно глубокий для обычных плагинов, но рекурсия требует осторожности.
